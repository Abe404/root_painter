# Releasing RootPainter

RootPainter ships several artifacts. Each is built by its own manually-triggered
GitHub Actions workflow (`workflow_dispatch`) and attached to a GitHub release.
The trainer is also published to PyPI.

## Version numbers

Bump both:

| Component | File | Field |
|-----------|------|-------|
| Painter (GUI) | `painter/src/main/python/about.py` | the `Version: X.Y.Z` line |
| Trainer (PyPI) | `trainer/pyproject.toml` | `version = "X.Y.Z"` |

The `.deb` and AppImage builds read the painter version from `about.py` with
`grep -oP 'Version: \K[0-9.]+'`, so nothing else needs editing for them.

## Artifacts

Workflows live in `.github/workflows/` and are run from the GitHub Actions tab
("Run workflow"). Each uploads an artifact, which you download and attach to the
release under the naming convention shown.

| Workflow | Runner | Produces | Released asset |
|----------|--------|----------|----------------|
| `build_ubuntu_22.yml` | ubuntu-22.04 | painter `.deb` | `RootPainter_<ver>_Ubuntu22.deb` |
| `build_windows.yml` | windows-latest | painter installer | `RootPainterInstaller_<ver>_windows.exe` |
| `build_osx.yml` | macOS | painter `.pkg` (arm64 + x86_64) | `RootPainter_<ver>_macOS_<arch>.pkg` |
| `build_workstation_ubuntu_cuda128.yml` (`variant: rtx50`) | ubuntu-24.04 | workstation AppImage, RTX 5000-series GPUs (sm_120) | `RootPainterWorkstation_<ver>_Ubuntu_CUDA128_RTX50.AppImage` |
| `build_workstation_ubuntu_cuda128.yml` (`variant: broad`) | ubuntu-24.04 | workstation AppImage, GTX 1660 – RTX 4090 (sm_75/80/86/89) | `RootPainterWorkstation_<ver>_Ubuntu_CUDA128_GTX1660_to_RTX4090.AppImage` |
| `build_workstation_windows.yml` | windows-latest | workstation installer | `RootPainterWorkstationInstaller_<ver>_windows.exe` |
| `build_workstation_macos.yml` | macos-14 | workstation `.pkg` | `RootPainterWorkstation_<ver>_macOS.pkg` |
| `build_torch_wheel.yml` | ubuntu-24.04 | custom minimal PyTorch wheels | **not** attached to the versioned release — hosted on their own prerelease tags (`torch-cu128-sm120-v1`, `torch-cu128-sm75-89-v1`) and consumed by the Ubuntu workstation build (see `CLAUDE.md`) |

A complete versioned release has these 8 assets (see `0.3.1`, `0.3.0`, `0.2.29`):

```
RootPainter_<ver>_Ubuntu22.deb
RootPainter_<ver>_macOS_arm64.pkg
RootPainter_<ver>_macOS_x86_64.pkg
RootPainterInstaller_<ver>_windows.exe
RootPainterWorkstation_<ver>_Ubuntu_CUDA128_RTX50.AppImage
RootPainterWorkstation_<ver>_Ubuntu_CUDA128_GTX1660_to_RTX4090.AppImage
RootPainterWorkstation_<ver>_macOS.pkg
RootPainterWorkstationInstaller_<ver>_windows.exe
```

The same builds can be run locally with the top-level scripts
(`build_workstation_ubuntu_cuda128.sh`, `build_workstation_mac.sh`,
`build_workstation_win.ps1`, `build_custom_torch.sh`) and, for the plain
painter, `painter/src/build/run_pyinstaller.py` then
`painter/src/build/make_deb_file.sh`.

## Publishing the trainer to PyPI

From `trainer/` (requires `build`, `twine`, and PyPI credentials):

```bash
./publish.sh        # uploads an sdist to TestPyPI
./publish.sh prod   # uploads to real PyPI (prompts for confirmation)
```

## Steps

1. Bump both versions and commit.
2. Run each workflow for this release.
3. Download each artifact, rename to the convention above.
4. Create the GitHub release / tag and upload the assets.
5. Publish the trainer with `trainer/publish.sh prod`.

## Notes

- The `.deb` control file is generated by `make_deb_file.sh`: `Architecture`
  comes from `dpkg --print-architecture`, `Version` from `about.py`, and the
  Qt/X runtime libraries are declared under `Depends:`. Do not replace it with a
  static control file (a hardcoded `Architecture: i386` previously caused `dpkg`
  to refuse the package on amd64 machines).
- Install the `.deb` with `sudo apt install ./RootPainter_<ver>_Ubuntu22.deb`
  so dependencies are pulled in. The package name is `rootpainter` (lowercase);
  query installed files with `dpkg -L rootpainter`.
